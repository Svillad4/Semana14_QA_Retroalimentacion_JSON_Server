# Lectura de clase: QA, usabilidad, accesibilidad y mejora continua

## 1. De construir a mejorar

En desarrollo de software no basta con crear una pagina o una funcionalidad. Despues de construir, llega una etapa igual de importante: revisar, probar, corregir y mejorar.

Esta etapa ayuda a responder preguntas como:

- El usuario entiende que debe hacer?
- El formulario protege la calidad de los datos?
- El sitio funciona en celular?
- La informacion se guarda correctamente?
- Hay errores en consola?
- Una persona con dificultades visuales podria usar la pagina?
- El codigo esta organizado para una siguiente version?

## 2. Usabilidad

La usabilidad mide que tan facil es usar un sistema. Un sitio usable es claro, directo y comprensible. No obliga al usuario a adivinar.

Ejemplos de buena usabilidad:

- botones con textos claros;
- mensajes de error especificos;
- formularios ordenados;
- navegacion visible;
- informacion separada por secciones;
- confirmaciones antes de acciones delicadas.

## 3. Accesibilidad

La accesibilidad busca que mas personas puedan usar el sitio. No se refiere solamente a discapacidad. Tambien incluye estudiantes con celulares sencillos, pantallas pequenas, conexiones lentas o poca experiencia digital.

Buenas practicas:

- usar `label` en formularios;
- usar `alt` en imagenes;
- conservar foco visible al navegar con teclado;
- usar contraste adecuado;
- no depender solamente del color para mostrar errores;
- escribir instrucciones claras.

## 4. Optimizacion

Optimizar significa mejorar el funcionamiento del proyecto. Puede incluir:

- eliminar codigo repetido;
- organizar archivos;
- reducir errores en consola;
- mejorar carga de datos;
- evitar validaciones debiles;
- mejorar responsive;
- escribir funciones mas claras.

## 5. QA como practica profesional

Cuando haces QA, piensas como usuario y como desarrollador. No solo preguntas "funciona?". Preguntas:

- funciona siempre?
- funciona con datos incorrectos?
- funciona en otro computador?
- funciona si JSON Server no esta activo?
- el usuario entiende el error?
- el equipo podria mantener este codigo despues?

Esa mirada es la que convierte una entrega basica en una entrega profesional.
